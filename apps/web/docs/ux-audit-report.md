### 3-1. 랜딩페이지 (`/`)

| 기준 | 현황 | 문제점 | 개선 제안 | 우선순위 |
|------|------|--------|----------|---------|
| 내비게이션 | URL 입력 → 분석 → 결과 단일 흐름 | - | 현재 흐름 적절 | P2 |
| 일관성 | 다크 테마 waide-dark 클래스 통일 | - | 양호 | P2 |
| 빈 상태 | 해당 없음 (정적 페이지) | - | - | - |
| 에러 핸들링 | URL 입력 유효성 검사 존재 | 분석 API 실패 시 사용자 안내 불명확 | try-catch에서 사용자 친화적 에러 메시지 표시 | P1 |
| 반응형 | 모바일 대응 적절 | - | 양호 | P2 |
| 정보 위계 | 히어로 → 분석 입력 → 소개 → 후기 순서 | - | 양호 | P2 |
| CTA | "무료 마케팅 점검 시작하기" 버튼 | CTA 이후 스크롤 하단의 전화/카카오 링크가 placeholder | 실제 연락처로 교체 필요 | P1 |
| **불필요 요소** | **하드코딩 데이터 다수** | **고객 후기 3건 = 더미 데이터, 통계 수치("200+ 매장") = 하드코딩, 전화번호 `02-0000-0000` = placeholder, 카카오톡 `_placeholder` URL** | **실데이터 연동 또는 제거. placeholder 연락처는 즉시 실제 값으로 교체** | **P0** |

**핵심 이슈:**
- 고객 후기 섹션 (3건)은 완전히 하드코딩된 더미 데이터. 실서비스 신뢰도에 직접 영향
- 하단 전화번호 `tel:02-0000-0000`, 카카오톡 `https://pf.kakao.com/_placeholder` — 사용자가 클릭하면 빈 페이지로 이동
- 통계 수치 ("200+ 매장이 신뢰하는") 근거 없는 하드코딩

| **빈 상태** | **SEO 진단, 키워드 순위 등 에이전트 결과 없을 수 있음** | **에이전트 결과 (경쟁사 분석, SEO 코멘트, 개선 플랜)가 없을 때 해당 섹션이 단순히 숨겨짐 — 사용자가 분석 범위를 알 수 없음** | **"AI 심층 분석은 유료 서비스에서 제공됩니다" 같은 upsell 메시지, 또는 "분석 중" 표시 추가** | **P0** |

| 반응형 | 대체로 양호 | 키워드 순위 테이블이 모바일에서 가로 스크롤 필요, 일부 섹션 좁은 화면에서 겹침 | 모바일용 카드 뷰 대체 또는 가로 스크롤 힌트 추가 | P1 |
| 정보 위계 | 점수 → SEO진단 → 키워드 → 전략 → 개선 순서 | - | 양호 | P2 |
| **CTA** | **전화/카카오/무료상담 3개 CTA** | **전화번호 `02-0000-0000`, 카카오 `_placeholder` = 작동 안 함. 가장 중요한 전환 페이지에서 CTA 불통** | **실제 연락처로 즉시 교체. 상담 신청 폼은 정상 작동 확인됨** | **P0** |
| **불필요 요소** | **placeholder 연락처** | **분석 결과 하단 "지금 바로 전화 상담" `02-0000-0000`, "카카오톡 상담" `_placeholder` — 실제 전환 페이지에서 가장 치명적** | **즉시 교체 또는 해당 CTA 버튼 제거** | **P0** |

**핵심 이슈:**
- 전환(Conversion) 직전 페이지에서 전화/카카오 CTA가 placeholder. 서비스 전체 퍼널 중 가장 치명적인 이슈
- 이미지 분석이 없는 경우(`imageAnalysisData.collection_failed`) 빈 상태 처리 불충분
- 1,375줄의 대형 단일 페이지 — 코드 유지보수성에도 영향

## 7. 데이터 연결 무결성

### 7-1. 브랜드 셀렉터 필터링

브랜드 셀렉터는 쿠키 기반(`selected_client`)으로 `getSelectedClientId()`를 통해 현재 선택된 클라이언트를 식별한다.

| 페이지 | clientId 전달 | 필터링 동작 | 판정 |
|--------|-------------|-----------|------|
| `/dashboard` | O | 선택 브랜드 KPI만 표시 | P2 |
| `/ops/keywords` | O | 선택 브랜드 키워드만 표시 | P2 |
| `/ops/analytics` | O | 선택 브랜드 SERP 데이터 | P2 |
| `/ops/recommendations` | O | 선택 브랜드 추천만 표시 | P2 |
| `/ops/accounts` | O | 선택 브랜드 계정 | P2 |
| `/ops/brands` | O | 전체 분석 목록 (셀렉터와 무관) | P2 |
| **`/ops/contents`** | **X** | **전체 클라이언트 콘텐츠 혼합 표시** | **P0** |
| `/ops/jobs` | O (간접) | 양호 | P2 |
| `/campaigns/plan` | O | 선택 브랜드 키워드/콘텐츠 | P2 |

**핵심 버그:** `/ops/contents` 페이지의 `getContents()` 호출에 `clientId` 파라미터가 누락되어, 브랜드 셀렉터와 무관하게 모든 클라이언트의 콘텐츠가 혼합 표시됨.

### 7-2. 캠페인 → SERP 데이터 플로우

```
캠페인 기획 (/campaigns/plan)
  │
  ├─ suggestKeywordsForClient() → keywords (status='suggested')
  │   └─ approveSuggestedKeyword() → status='active'
  │
  ├─ addManualKeyword() → keywords (status='active')
  │
  └─ triggerContentGeneration({clientId, keywords, styleRefs, contentType, instructions})
       │
       ├─ jobs INSERT (CONTENT_CREATE, PENDING)
       │   └─ input_payload: {keyword, clientId, contentType, styleRefs, instructions}
       │
       └─ processContentJobs()
            │
            └─ generateContentV2()
                 ├─ getBenchmark() → content_benchmarks (7일 캐시)
                 ├─ createContentV2() → COPYWRITER v2 → contents INSERT
                 ├─ runQcV2() → QC v2 → contents.metadata UPDATE
                 └─ runRewriteLoop() → FAIL 시 재작성(최대 2회)
                      │
                      └─ contents (publish_status: 'draft' → 'approved')
                           │
                           [에디터에서 수동 발행]
                           │
                           └─ updatePublishedUrl(contentId, url)
                                ├─ is_tracking = true
                                ├─ publish_status = 'published'
                                └─ published_url = URL
                                     │
                                     [Vercel Cron /api/cron/serp — 매일]
                                     │
                                     ├─ collectSerpAll() → serp_results
                                     ├─ keyword_visibility UPDATE
                                     └─ daily_visibility_summary 집계
```

**판정: P2 (양호)** — 캠페인 기획 → 콘텐츠 생성 → 발행 → SERP 추적 플로우가 end-to-end로 연결되어 있음. Job→Content 연결은 application-level (JSON payload 내 참조)이며 DB FK는 아니지만 기능적으로 작동함.

**참고: SERP 추적 지연** — 발행 URL 입력 후 SERP 수집은 다음 크론 실행(매일)까지 최대 24시간 대기. `updatePublishedUrl()` 시점에 즉시 SERP 체크를 트리거하면 사용자 경험 향상 가능.

### 7-3. 포털 vs 어드민 데이터 소스

| 데이터 | 어드민 소스 | 포털 소스 | 동일 테이블? | 판정 |
|--------|-----------|----------|------------|------|
| 키워드 목록 | `keywords` | `keywords` (client_id 필터) | O | P2 |
| 콘텐츠 | `contents` | `contents` (client_id 필터) | O | P2 |
| 순위 데이터 | `keyword_visibility`, `serp_results` | `keyword_visibility` (client_id 필터) | O (포털은 kv만 사용) | P2 |
| 분석 결과 | `brand_analyses` | `brand_analyses` (client_id 필터) | O | P2 |
| 리포트 | `report_deliveries` | `report_deliveries` (client_id 필터) | O | P2 |
| 구독 | `subscriptions` | `subscriptions` (client_id 필터) | O | P2 |

**판정: P2 (양호)** — 포털과 어드민이 동일 테이블을 사용하되, 포털은 항상 `client_id` 필터를 적용하여 자기 데이터만 접근. 데이터 소스 불일치 없음.

**주의점:** `serp_results` 테이블에는 `client_id` 컬럼이 없음. 포털에서 순위 데이터 접근 시 반드시 `keyword_visibility` 테이블(client_id 보유)을 사용해야 함 — 현재 코드에서 올바르게 처리됨.

## 8. 역할별 사용자 플로우

### 8-1. 고객 포털 사용자 (Client Owner)

**핵심 워크플로우:** 로그인 → 대시보드 확인 → 키워드 관리 → 콘텐츠 확인 → 리포트 조회

| 단계 | 액션 | 현재 가능한 흐름 | 끊기는 지점 | 우선순위 |
|------|------|----------------|-----------|---------|
| 1 | 로그인 | `/login` → 이메일 입력 → Supabase Auth → `/portal` 리다이렉트 | client_id 미연결 사용자는 대기 페이지 (PortalPendingPage) 표시 — 정상 처리됨 | P2 |
| 2 | 대시보드 확인 | KPI 4종 + 브랜드 요약 + 개선 제안 + 최근 활동 | 신규 고객 시 모든 데이터 0/empty — "서비스 준비 중" 가이드 부재 | P1 |
| 3 | AI 추천 키워드 승인 | `/portal/keywords` → AI추천 탭 → [승인] 버튼 | 정상 작동. keyword status 'suggested' → 'active' | P2 |
| 4 | 콘텐츠 확인 | `/portal/contents` → 상태 필터 → 상세 보기 | 정상 작동. QC 결과, 재작성 이력 확인 가능 | P2 |
| 5 | 리포트 조회 | `/portal/reports` → 월 선택 → 차트/순위/PDF | 정상 작동. PDF 다운로드 가능 | P2 |
| 6 | 설정 변경 | `/portal/settings` → 프로필/비밀번호/구독 | 프로필+비밀번호 변경 가능. 구독 정보는 읽기 전용 (변경은 어드민에서) | P2 |

**종합 판정: P1** — 핵심 5개 워크플로우 모두 기능적으로 작동하나, 신규 고객 온보딩 경험(빈 데이터 상태)에서 가이드 부족.

### 8-2. 운영자/영업사원 (Operator / Sales)

**핵심 워크플로우:** 고객 등록 → 키워드 확보 → 콘텐츠 생성 → 발행 → SERP 추적 → CRM 관리

| 단계 | 액션 | 현재 가능한 흐름 | 끊기는 지점 | 우선순위 |
|------|------|----------------|-----------|---------|
| 1 | 고객 등록 | `/ops/clients` → 신규 고객 등록 | DB에서 직접 INSERT 필요 (UI에 생성 폼 없음) — 실제로는 분석 결과에서 CRM 연동 시 자동 생성 | P1 |
| 2 | 키워드 확보 | `/campaigns/plan` → AI 추천 또는 수동 입력 | 정상 작동. AI 추천 → 승인 → active 흐름 완성 | P2 |
| 3 | 콘텐츠 생성 지시 | `/campaigns/plan` → 키워드 선택 → [생성 지시] | 정상 작동. CONTENT_CREATE Job 생성 → processContentJobs() 실행 | P2 |
| 4 | 콘텐츠 검토/편집 | `/ops/contents/[id]` → 마크다운 확인 → QC 결과 확인 | 정상 작동 | P2 |
| 5 | 마크다운 복사 + 블로그 발행 | 에디터에서 [마크다운 복사] → 블로그에 수동 붙여넣기 | **수동 프로세스** — 자동 발행(Phase D) 미구현. 운영자가 블로그 로그인 → 붙여넣기 필요 | P1 |
| 6 | 발행 URL 입력 | `/ops/contents/[id]` → 발행 URL 입력 → is_tracking=true | 정상 작동 | P2 |
| 7 | SERP 추적 확인 | `/ops/clients/[id]` → 순위 탭 | SERP 크론 실행 후 데이터 표시. **최대 24시간 대기** (발행 직후 즉시 확인 불가) | P1 |
| 8 | CRM 관리 | `/ops/analysis-logs` → 상태 변경 / 코멘트 / 영업사원 할당 | 정상 작동. lead_status 파이프라인 완성 | P2 |

**종합 판정: P1** — 8단계 워크플로우 중 5단계(수동 발행)와 7단계(SERP 지연)에서 마찰 존재. 자동 발행은 Phase D로 계획됨.
